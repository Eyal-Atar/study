# 19.1-03 â€” Implement First-Time Achievement Badge Unlocks

## Context

The current `_BADGE_CRITERIA` list in `backend/gamification/utils.py` only
covers streak length, level, and XP total. There are no "first-time event"
badges for:
- Completing the first task ever
- Logging in for the first time (first login registered)
- Achieving a 7-day streak for the first time (`week_streak`)
- Having a streak broken (`streak_broken_once` â€” learning-moment badge)

These require a counter or flag that doesn't exist yet. The `user_xp` table
has no `tasks_completed` column. The badge system uses a lambda list checked
after every `award-xp` call, but there's no input for task completion count.

This plan adds:
1. `tasks_completed` INTEGER column to `user_xp` (migration-safe, default 0)
2. Increment `tasks_completed` inside `update_user_xp` (called from `award-xp`)
3. Four new badge criteria using the existing lambda pattern
4. Pass incremented `tasks_completed` value into `check_and_award_badges` via
   the `user_xp_row` dict so lambdas can reference it

The `first_login` badge is a special case: it fires on the very first call to
`update_streak` (when the `user_streaks` row is created). The badge check
already runs inside `login-check` after `update_streak`, so we only need to
add the criterion.

### Task 2 â€” Increment `tasks_completed` in `update_user_xp`

**File:** `backend/gamification/utils.py`

Inside `update_user_xp`, after the level calculation and before the `UPDATE`
statement, increment `tasks_completed` by 1 every time XP is awarded (i.e.,
for every block marked done). This is acceptable as a proxy for "tasks
completed" since `award-xp` is idempotent and only fires once per block.

Changes:

1. In the `INSERT INTO user_xp` branch (new user), include `tasks_completed`:
```python
db.execute(
    "INSERT INTO user_xp (user_id, total_xp, current_level, daily_xp, daily_xp_date, tasks_completed) VALUES (?, 0, 1, 0, ?, 0)",
    (user_id, today),
)
```

2. Read `tasks_completed` from the fetched row:
```python
tasks_completed = (row["tasks_completed"] or 0) + 1
```

3. Include `tasks_completed` in the `UPDATE` statement:
```python
db.execute(
    """UPDATE user_xp
       SET total_xp = ?, current_level = ?, daily_xp = ?, daily_xp_date = ?, tasks_completed = ?
       WHERE user_id = ?""",
    (new_total_xp, new_level, new_daily_xp, today, tasks_completed, user_id),
)
```

4. Include `tasks_completed` in the returned dict:
```python
return {
    "total_xp": new_total_xp,
    "current_level": new_level,
    "daily_xp": new_daily_xp,
    "level_up": new_level > prev_level,
    "tasks_completed": tasks_completed,
}
```

**Expected outcome:** Each `award-xp` call increments the counter. The
`check_and_award_badges` call that follows in `routes.py` will receive
`tasks_completed` in the `user_xp_row` dict.

### Task 3 â€” Add four first-time achievement badge criteria

**File:** `backend/gamification/utils.py`, `_BADGE_CRITERIA` list

Append four new entries to `_BADGE_CRITERIA`:

```python
_BADGE_CRITERIA = [
    # --- existing entries unchanged ---
    ("iron_will_7",     lambda xp, s: s["current_streak"] >= 7),
    ("iron_will_30",    lambda xp, s: s["current_streak"] >= 30),
    ("iron_will_100",   lambda xp, s: s["current_streak"] >= 100),
    ("knowledge_seeker_5",  lambda xp, s: xp["current_level"] >= 5),
    ("knowledge_seeker_10", lambda xp, s: xp["current_level"] >= 10),
    ("knowledge_seeker_25", lambda xp, s: xp["current_level"] >= 25),
    ("knowledge_seeker_50", lambda xp, s: xp["current_level"] >= 50),
    ("xp_1000",  lambda xp, s: xp["total_xp"] >= 1000),
    ("xp_5000",  lambda xp, s: xp["total_xp"] >= 5000),
    ("xp_10000", lambda xp, s: xp["total_xp"] >= 10000),
    # --- new: first-time achievement badges ---
    # Fires on first ever task completed (tasks_completed key present from 19.1-03)
    ("first_task",         lambda xp, s: (xp.get("tasks_completed") or 0) >= 1),
    # Fires on first ever login (current_streak >= 1 and user_streaks row just created)
    # Since first_login_today is not in xp row, use streak == 1 and longest == 1 as proxy
    ("first_login",        lambda xp, s: s["current_streak"] >= 1 and s["longest_streak"] == 1),
    # Fires when user reaches a 7-day streak (same event as iron_will_7; this is an
    # "achievement moment" badge distinct from the milestone badge)
    ("week_streak",        lambda xp, s: s["current_streak"] >= 7),
    # Fires when streak is broken (streak_broken flag is True in streak_row)
    ("streak_broken_once", lambda xp, s: bool(s.get("streak_broken", False))),
]
```

Notes:
- `first_login` uses `longest_streak == 1` as a proxy for "never had a streak
  before." This is only True on the very first day. Once any day-2 login
  happens, `longest_streak` becomes 2 and the badge would already be earned.
  The `INSERT OR IGNORE` in `check_and_award_badges` makes this idempotent.
- `week_streak` and `iron_will_7` both fire at streak >= 7. Two distinct badges
  are awarded simultaneously. This is intentional (one is a milestone badge,
  one is an achievement moment badge).
- `streak_broken_once` fires on the day after a missed day. `streak_row` passed
  from `login-check` already contains `streak_broken=True` in that scenario.

**Expected outcome:** These four badges are checked and potentially awarded
on every `award-xp` call and every `login-check` call, using the same
`check_and_award_badges` function already in place.

### Task 4 â€” Verify `routes.py` passes updated `xp_result` to `check_and_award_badges`

**File:** `backend/gamification/routes.py`, `award_xp` endpoint, lines 173â€“177

After Task 2, `update_user_xp` returns `tasks_completed` in its dict.
`check_and_award_badges(db, user_id, xp_result, streak_row)` already receives
this dict as `user_xp_row`. The new lambda `xp.get("tasks_completed")` will
resolve correctly.

No code change needed in `routes.py` for the `award-xp` endpoint.

For `login-check`, the flow is: `check_and_award_badges(db, user_id, xp_row, streak_result)`.
`xp_row` is fetched from `_get_xp_row()` which runs `SELECT total_xp, current_level,
daily_xp, daily_xp_date FROM user_xp`. After Task 1/2, this row now also has
`tasks_completed`.

Update `_get_xp_row` to include `tasks_completed` in both the SELECT and the
fallback dict:

```python
def _get_xp_row(db, user_id: int) -> dict:
    row = db.execute(
        "SELECT total_xp, current_level, daily_xp, daily_xp_date, tasks_completed FROM user_xp WHERE user_id = ?",
        (user_id,),
    ).fetchone()
    if row is None:
        return {"total_xp": 0, "current_level": 1, "daily_xp": 0, "daily_xp_date": None, "tasks_completed": 0}
    return dict(row)
```

**File to modify:** `backend/gamification/routes.py`, `_get_xp_row` function.

**Expected outcome:** `first_task` badge check inside `login-check` has access
to `tasks_completed` and correctly awards the badge if the user has completed
at least one block.
