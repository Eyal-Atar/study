# Plan 21-01: Backend Foundation & Legacy Cleanup

Modernize the backend to support material-based task anchoring, user profiling preferences, and a unified onboarding submission endpoint.

<task>
<name>DB Migration (Onboarding & Profiling)</name>
<files>
- backend/server/database.py
- backend/users/schemas.py
</files>
<action>
Add the following fields to the `users` table in `database.py`:
- `onboarding_completed` (INTEGER DEFAULT 0)
- `study_hours_preference` (TEXT) - Store as JSON string (e.g., ["morning", "afternoon"])
- `buffer_days` (INTEGER DEFAULT 1) - Default 1, options 0, 1, 2.
Update `UserResponse` and create a new `UserOnboardRequest` schema in `schemas.py`.
</action>
<verify>
Verify `/users/me` returns the new fields and the DB schema is correctly updated.
</verify>
<done>
Profiling and onboarding fields exist in the database and API schemas.
</done>
</task>

<task>
<name>Unified Onboarding Endpoint</name>
<files>
- backend/brain/routes.py
- backend/brain/schemas.py
</files>
<action>
Create a new POST `/users/onboard` endpoint in `routes.py`. This endpoint should:
- Update user profiling (study_hours, buffer_days).
- Transactionally create multiple exams and associate uploaded files.
- Trigger the initial roadmap generation.
- Set `onboarding_completed = 1`.
</action>
<verify>
Test the endpoint with a mock multipart payload containing profile, exam data, and files.
</verify>
<done>
Unified onboarding submission endpoint is functional.
</done>
</task>

<task>
<name>Scheduler & Anchoring Updates</name>
<files>
- backend/brain/exam_brain.py
- backend/brain/scheduler.py
</files>
<action>
Update `exam_brain.py` (Auditor) and `scheduler.py` (Enforcer) to:
- Anchor Simulations to final days and Reviews to day 1.
- **Dynamic Naming Logic:** If `past_exam` files exist, rename Simulation tasks to "Simulation: [Exam Filename]".
- **Nudge Mechanism:** If no `past_exam` is uploaded, still create "Full Simulation" tasks as a reminder.
- Respect `study_hours_preference` (Morning, Afternoon, Night) and `buffer_days` (0-2).
</action>
<verify>
Verify generated schedules show specific "Simulation: [Name]" when files exist and generic "Full Simulation" otherwise.
</verify>
<done>
Scheduler respects profiling constraints, material-based anchoring, and dynamic task naming.
</done>
</task>
